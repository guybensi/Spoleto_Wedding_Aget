PRE-IMPLEMENTATION ONBOARDING GUIDE

This is the external setup checklist to complete before writing code for the Spoleto travel assistant. It assumes the current architecture stays the same: OpenClaw as the main runtime, native WhatsApp integration, Claude models, Python for ingestion support, Docker for packaging, and a production target of one private deployment.

RECOMMENDED ORDER

1. Decide hosting and ownership boundaries.
2. Create core accounts and billing.
3. Prepare the developer workstation.
4. Set up GitHub project controls.
5. Prepare WhatsApp numbers and groups.
6. Prepare source documents and data hygiene rules.
7. Decide secrets handling.
8. Prepare staging and production environments.
9. Run safe test onboarding before real guests are invited.

8. DEPLOYMENT OPTIONS COMPARISON

LOCAL MACHINE
Advantages: fastest start, cheapest, easiest QR login, best debugging.
Disadvantages: poor uptime, tied to your laptop, sleep and restart risk, weaker production hygiene.
Use for: local development only.
Recommendation: not for production.

RASPBERRY PI
Advantages: cheap, low power, self-hosted, always-on if configured well.
Disadvantages: weaker CPU and I/O, SD-card wear unless using SSD, more hands-on ops, less forgiving for OCR and multi-service Docker stacks.
Use for: hobby staging or cost-sensitive personal deployment.
Recommendation: only if you already operate Pi infrastructure and use SSD plus good backup discipline.
Docs: https://www.raspberrypi.com/documentation/computers/getting-started.html

VPS
Advantages: best balance of reliability, cost, control, SSH access, persistent storage, and easy Docker support.
Disadvantages: you manage OS patching, firewalling, backups, and host security.
Use for: primary production recommendation.
Recommendation: best fit for this project.

RAILWAY
Advantages: fast PaaS setup, good DX, easy variables and volumes, simpler than raw cloud.
Disadvantages: less natural fit for long-lived QR-linked stateful WhatsApp automation than a plain VPS, more abstraction during debugging, less control over host-level concerns.
Use for: fast prototypes or if you strongly prefer managed deployment.
Recommendation: acceptable, but not my first choice here.
Docs: https://docs.railway.com/

FLY.IO
Advantages: strong networking, good Docker support, mature docs, flexible runtime options.
Disadvantages: more operational complexity than this project needs, stateful volumes and regional placement are more involved, steeper learning curve for a small private assistant.
Use for: teams already comfortable with Fly.
Recommendation: viable, but not optimal for a first build.
Docs: https://fly.io/docs/

RENDER
Advantages: polished managed platform, Docker support, disks, logs, managed operational surface.
Disadvantages: still more indirect than a plain VPS for QR-based long-lived messaging runtimes and host-level recovery.
Use for: managed-platform preference with lower ops burden.
Recommendation: strongest managed-platform alternative to a VPS.
Docs: https://render.com/docs

DIGITALOCEAN
Advantages: simple droplets, clear docs, common Linux admin path, easy SSH-key workflows.
Disadvantages: if using a Droplet, you still manage everything yourself; if using App Platform, you inherit managed-platform constraints.
Use for: VPS deployment, especially if you already use DigitalOcean.
Recommendation: good VPS provider choice.
Docs: https://docs.digitalocean.com/products/droplets/

BEST OPTION FOR THIS PROJECT
Best choice: a small Linux VPS with Docker Compose.
Why: this assistant is stateful, QR-linked, long-running, private, and operationally simple enough that a VPS gives you the cleanest control without overengineering. If you want managed hosting, Render is the best alternative from this list.
Useful docs:
- OpenClaw Docker: https://docs.openclaw.ai/install/docker
- Railway docs: https://docs.railway.com/
- Fly.io docs: https://fly.io/docs/
- Render docs: https://render.com/docs
- DigitalOcean Droplets: https://docs.digitalocean.com/products/droplets/

9. SECRETS MANAGEMENT CHECKLIST

1. Choose a secret manager approach.
Why: secrets must not live in source code, prompt files, or plaintext shared docs.
Mandatory: Yes.
When: Before any key is created.
Recommendation: local .env only for development; production secrets in host environment or a managed secret store.
Docs: https://docs.openclaw.ai/gateway/configuration
Docs: https://docs.github.com/en/actions/security-guides/encrypted-secrets

2. Inventory all expected secrets.
Why: you should know what needs rotation, backup protection, and access control.
Mandatory: Yes.
When: Before implementation.
Expected secrets in this architecture:
- ANTHROPIC_API_KEY
- OPENCLAW_GATEWAY_TOKEN or gateway password
- WhatsApp linked-device auth state files under OpenClaw credentials
- Search provider API key if used
- Weather provider API key if used
- Currency or flight provider API key if used later
- Hook token if webhooks are enabled
- GitHub Actions deployment secrets
- VPS SSH private key
- Tailscale auth key if Tailscale is used
- Backup encryption passphrase or key
- Optional SMTP or API tokens for alerts
- Optional Gmail or email integration OAuth credentials if email search is added later

3. Define who can access which secret.
Why: local dev access should not equal production access.
Mandatory: Yes.
When: Before team collaboration or CI and CD.

4. Define rotation rules.
Why: tokens for gateways, providers, and SSH access should be replaceable without ad hoc cleanup.
Mandatory: Recommended.
When: Before production.

5. Define backup and restore treatment for secret-bearing files.
Why: WhatsApp auth state and OpenClaw state are sensitive but operationally critical.
Mandatory: Yes.
When: Before production backups start.

11. TESTING BEFORE REAL GUESTS

1. Use a separate test WhatsApp group.
Why: never test reply behavior, reminders, or failures in the real guest group first.
Mandatory: Yes.
When: Before first WhatsApp integration test.

2. Use either a dedicated staging WhatsApp number or a cloned staging state.
Why: you need to test relinking, disconnects, and group behavior without risking the production identity.
Mandatory: Strongly recommended.
When: Before staging rollout.

3. Use fake or sanitized trip documents first.
Why: this lets you test ingestion, OCR, and prompting without exposing guest data too early.
Mandatory: Yes.
When: Before the first end-to-end test.

4. Define test scenarios before implementation.
Why: safe testing is much easier when expected answers are known in advance.
Mandatory: Yes.
When: Before the first integrated prototype.
Minimum scenarios:
- itinerary lookup
- hotel question
- translation
- pharmacy lookup
- wrong-date conflict
- missing-info refusal
- reminder send
- reconnect recovery

5. Test reminder delivery only in staging first.
Why: scheduled messages are the fastest way to create embarrassing failures in a real group.
Mandatory: Yes.
When: Before any cron use in production.

6. Run failure drills.
Why: you need operational confidence, not only happy-path success.
Mandatory: Recommended.
When: Before production.
Drills:
- lost WhatsApp auth
- bad OCR extraction
- missing API key
- gateway restart
- corrupted config
- provider outage

7. Prepare a rollback and kill-switch procedure.
Why: if the bot misbehaves in the guest group, you need a fast stop path.
Mandatory: Yes.
When: Before inviting real guests.
Recommendation: documented command to stop gateway, remove group access, and disable reminders.
